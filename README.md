# 对抗审查 Skill

> 每次任务完成时自动派遣独立智能体，以「找茬者」立场审查任务完成度、漏洞隐患和优化空间。

## 核心思想

**自我审查永远有盲区。** 大模型输出后倾向于「自我维护」，不会主动否定自己。解决方案：派一个没有执行上下文的独立智能体去专门找茬。

- 全新上下文，无历史包袱
- 明确对抗立场，不会「过度谄媚」
- 强制找茬要求，必须输出问题

## 三大审查维度

### 1. 任务完成度验证

任务真的完成了吗？

- 用户原话中的每一项要求是否都被满足？
- 执行计划中的每一个步骤是否都已完成？
- 交付物是否齐全？
- 涉及实际操作（发帖、部署、提交）时，是否有执行证据？

### 2. 漏洞与隐患挖掘

有哪些隐藏的问题？

- **模糊点识别**：「优化一下」→ 优化到什么程度？列出不同理解导致的执行差异
- **范围蔓延**：需求是否可能在执行中不断扩大？
- **质量风险**：完成标准是否明确？
- **依赖风险**：是否依赖未提及的外部条件？
- **验证风险**：如何确认任务真正完成？

### 3. 优化建议

哪里可以做得更好？

- 任务定义是否可以更清晰？
- 是否遗漏了隐含需求？
- 是否有更高效的执行方式？

## 使用示例

### 示例一：代码任务

**用户原话**：「帮我写一个用户登录功能」

**对抗审查报告**：

```
### 一、完成度：部分完成 70%

| 要求 | 状态 | 问题 |
|------|------|------|
| 用户登录功能 | ✅ | 已实现 |
| 密码加密 | ❌ | 未提及是否需要加密存储 |
| 错误处理 | ❓ | 未说明异常情况如何处理 |

未达标项：
1. 密码存储方式未定义 — 影响：可能明文存储 — 建议：明确使用 bcrypt 加密

### 二、模糊点与隐患

模糊点：
1. 「登录功能」→ 仅账号密码？还是支持手机号/第三方登录？— 建议：明确登录方式

隐患：
1. [安全风险] 未提及密码加密 — 触发场景：数据库泄露 — 后果：用户密码泄露 — 建议：强制 bcrypt

### 三、优化建议

1. [高优先级] 添加密码加密 — 理由：基本安全要求
2. [中优先级] 添加登录失败次数限制 — 理由：防止暴力破解
```

### 示例二：文档任务

**用户原话**：「帮我写一份项目介绍文档」

**对抗审查报告**：

```
### 一、完成度：存疑 60%

| 要求 | 状态 | 问题 |
|------|------|------|
| 项目介绍文档 | ✅ | 已生成 |
| 文档内容 | ❓ | 无法验证内容是否准确 |
| 文档格式 | ❌ | 未说明需要什么格式（Markdown/Word/PDF） |

未达标项：
1. 文档格式未定义 — 影响：可能生成错误格式 — 建议：明确输出格式

### 二、模糊点与隐患

模糊点：
1. 「项目介绍」→ 技术介绍？产品介绍？还是融资 BP？— 建议：明确受众和用途

### 三、优化建议

1. [高优先级] 明确文档受众和用途 — 理由：不同受众需要完全不同的内容
```

## 触发方式

- **自动触发**：每次任务完成时自动执行（无需用户操作）
- **手动触发**：说「对抗审查」「找茬」「审查任务」即可

## 注意事项

- 只传用户原话和计划原文给审查智能体，不传执行过程
- 审查智能体在完全隔离的环境中工作
- 至少找出 5 个问题，除非真的无懈可击
