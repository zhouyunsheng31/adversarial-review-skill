# 对抗审查 Skill (Adversarial Review)

## 名称
`adversarial-review`

## 描述
每次完成任务后，自动派一个独立智能体以「找茬者」身份进行对抗性审查，从三个核心维度全面检验任务质量，有效提升任务鲁棒性和通过率。

## 三大审查维度

### 1. 任务完成度验证 ✅
**核心问题：任务真的完成了吗？**

- **逐条核对**：将任务描述拆成 checklist，逐条检查是否完成
- **实际执行验证**：
  - 发帖任务 → 要求提供社媒链接/截图证明
  - 代码任务 → 要求实际运行测试
  - 文档任务 → 检查文件是否真实存在
  - 分析任务 → 验证数据来源可靠性
- **判定标准**：只要有任何一项要求没执行、交付物缺失、无法提供证明，都不算"完成"

### 2. 漏洞与隐患挖掘 🔍
**核心问题：有哪些隐藏的问题和重大隐患？**

#### 任务描述不清的识别
- 范围模糊："优化一下"——优化到什么程度？
- 标准缺失："做好"——好的标准是什么？
- 边界不清："处理数据"——多少数据？什么格式？
- 识别模糊点后，列出不同理解下的执行差异

#### 重大隐患挖掘
- **系统性风险**：单点故障、依赖风险、扩展性问题
- **安全风险**：数据泄露、权限漏洞、注入攻击
- **业务风险**：合规问题、用户体验、成本失控
- **技术债务**：临时方案永久化、硬编码、无文档

### 3. 优化建议 💡
**核心问题：哪些地方可以做得更好？**

- **效率优化**：流程简化、自动化机会、并行化
- **质量提升**：错误处理、用户体验、可观测性
- **架构优化**：解耦、抽象、扩展点、回滚能力
- **最佳实践**：行业标准、工具选择、命名规范

## 使用场景
- 代码开发完成后需要质量把关
- 文档撰写后需要第三方视角审视
- 方案设计后需要挑刺验证
- 社媒发帖后需要确认是否真的提交
- 任何需要「第二双眼睛」审查的任务

## 核心特性
1. **独立智能体**：启动全新对话，无主任务上下文污染
2. **对抗性定位**：明确告知「你是来找茬的，不是来帮忙的」
3. **不谄媚**：要求刻意寻找漏洞，不会因为「还行」就放行
4. **三大维度**：完成度验证 + 漏洞挖掘 + 优化建议
5. **结构化输出**：清晰的表格和分级报告

## 工作流程
1. 用户完成某项任务后，调用此 Skill
2. Skill 自动提取本次任务的所有输出内容
3. 启动独立智能体，注入「对抗审查者」角色
4. 智能体从三个维度对任务输出进行全方位审查
5. 输出结构化审查报告

## 输出格式示例

```markdown
# 对抗审查报告

## 【任务完成度】✅/❌ 验证结果
### 完成状态：[完成/部分完成/未完成]
### 核对清单：
| 任务要求 | 是否完成 | 证据 | 问题说明 |
|---------|---------|------|---------|
| 要求1 | ✅ | [链接/截图] | - |
| 要求2 | ❌ | 无 | 未实际执行 |

### 未达标项：
1. **问题**：xxx
   - **期望**：应该...
   - **实际**：实际...
   - **影响**：会导致...
   - **修复建议**：需要...

---

## 【漏洞与隐患】🔍 深度挖掘
### 任务描述不清
1. **模糊点**："优化一下"
   - **可能理解1**：稍微改改格式
   - **可能理解2**：重写整个逻辑
   - **建议澄清**：明确优化目标和验收标准

### 重大隐患
1. **隐患类型**：[安全风险]
   - **问题描述**：用户输入未验证
   - **触发条件**：输入特殊字符
   - **潜在后果**：SQL注入
   - **建议措施**：增加输入校验

---

## 【优化建议】💡 可以更好
### 高优先级
1. **优化方向**：增加错误处理
   - **当前问题**：异常时直接崩溃
   - **优化方案**：try-catch + 友好提示
   - **预期收益**：提升用户体验
```

## 使用方式
在 SOLO 对话中，直接说：
- "帮我审查一下刚才的代码"
- "启动对抗审查"
- "用找茬模式检查这个方案"
- "检查任务是否真的完成了"

## 注意事项
- 此 Skill 不会自我审查（避免上下文污染）
- 每次审查由独立智能体执行
- 不会因为「看起来没问题」就停止审查
- 要求提供实际执行证据（链接、截图、测试结果等）
